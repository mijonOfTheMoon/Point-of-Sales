4.3 Anggota 3: A. Wildan Hula Aufa

4.3.1 Modul yang Dikerjakan

Nama Modul: Pengeluaran

4.3.2 Deskripsi Fitur

Fitur ini digunakan untuk mencatat pengeluaran toko yang nantinya akan langsung mengurangi saldo sebuah kas. Tiap pengeluaran menyimpan deskripsi, nominal, kas sumber dana, serta status. Karena tiap pengeluaran selalu menunjuk ke satu kas, modul ini akan bergantung pada data kas yang aktif saat pengguna mengisi formnya. Pencatatan maupun pembatalannya berjalan lewat RPC supaya pengurangan dan pengembalian saldo kas tetap dihitung oleh database.

Fitur yang dikerjakan meliputi:
1. Menampilkan daftar pengeluaran beserta jumlah catatan dan total nominal yang masih aktif
2. Mencatat pengeluaran baru lewat bottom sheet dengan memilih kas sumber dana
3. Membatalkan pengeluaran yang sudah tercatat
4. Mencari serta mengurutkan pengeluaran berdasarkan deskripsi, nominal, status, atau yang paling baru

4.3.4 Kode Model

Kotlin

@Serializable
data class Expense(
    val id: String? = null,
    val description: String,
    @SerialName("total")
    val amount: Double,
    val kas_id: String,
    val status: String = "active",
    val created_at: String? = null,
    val updated_at: String? = null
) {
    val is_cancelled: Boolean
        get() = status == "cancelled"
}

4.3.5 Penjelasan Kode Model

Model Expense akan memetakan satu baris tabel expense di Supabase. Ada satu hal yang membedakannya dari model lain, yaitu anotasi SerialName pada field amount. Di database, kolomnya bernama total, tetapi nama itu terasa kurang jelas saat dipakai di aplikasi. Lewat SerialName total, kolom total tadi akan tetap dipetakan ke property amount di Kotlin, sehingga nama di sisi aplikasi bisa lebih bermakna tanpa harus mengubah skema database. Field kas_id menjadi penghubung ke kas sumber dana, jadi tiap pengeluaran selalu tahu saldo kas mana yang harus berkurang. Field status menyimpan keadaan pengeluaran dengan nilai default active.

Di bagian bawah ada property turunan is_cancelled yang nilainya tidak disimpan di database, melainkan dihitung saat diakses lewat blok get. Property ini akan bernilai true hanya ketika status-nya sama dengan cancelled. Dengan adanya property turunan ini, layar tidak perlu lagi membandingkan string status secara manual di banyak tempat, dan cukup membaca is_cancelled saja. Sama seperti model lain, field id, created_at, dan updated_at dibuat nullable karena nilainya akan diisi sendiri oleh database.

4.3.6 Kode Repository

Kotlin

class ExpenseRepository {
    private val postgrest = SupabaseClientProvider.client.postgrest

    suspend fun getExpenses(): List<Expense> = withContext(Dispatchers.IO) {
        postgrest["expense"].select().decodeList<Expense>()
    }

    suspend fun createExpense(description: String, amount: Double, kasId: String) = withContext(Dispatchers.IO) {
        postgrest.rpc(
            function = "create_expense",
            parameters = buildJsonObject {
                put("p_description", description)
                put("p_total", amount)
                put("p_kas_id", kasId)
            }
        )
    }

...

}

4.3.7 Penjelasan Kode Repository

ExpenseRepository menangani seluruh komunikasi data pengeluaran ke Supabase. Di bagian atas, repository menyimpan referensi postgrest dari SupabaseClientProvider supaya dipakai bersama oleh semua function. Seperti repository lain, function di sini ditandai suspend sehingga hanya bisa dipanggil dari coroutine dan bisa ditunda tanpa memblokir thread. Pembacaan data ditangani getExpenses yang menjalankan select pada tabel expense lalu men-decode hasilnya menjadi List<Expense>. Pencatatan pengeluaran baru dilakukan lewat createExpense yang memanggil RPC create_expense. Perhatikan bahwa nominalnya dikirim lewat parameter p_total mengikuti nama kolom di database, bukan amount, supaya cocok dengan parameter yang diharapkan RPC. Parameter p_kas_id menentukan kas mana yang saldonya akan berkurang begitu pengeluaran tercatat. Untuk pembatalan, repository memanggil RPC cancel_expense lewat cancelExpense supaya database yang bertugas mengembalikan saldo kas terkait, bukan aplikasi. Seluruh operasi tadi dibungkus withContext Dispatchers.IO supaya pekerjaan jaringannya berjalan di thread pool khusus input output, sehingga main thread tetap bebas dan tampilan tidak ikut macet.

4.3.8 Kode ViewModel

Kotlin

class ExpenseViewModel(private val repository: ExpenseRepository = ExpenseRepository()) : ViewModel() {

    private val _uiState = MutableStateFlow<ExpenseUiState>(ExpenseUiState.Loading)
    val uiState: StateFlow<ExpenseUiState> = _uiState.asStateFlow()

    init {
        loadExpenses()
    }

...

    fun createExpense(description: String, amount: Double, kasId: String) {
        if (kasId.isBlank()) {
            _uiState.value = ExpenseUiState.Error("Kas id is missing")
            return
        }
        viewModelScope.launch {
            try {
                repository.createExpense(description, amount, kasId)
                loadExpenses()
            } catch (e: Exception) {
                _uiState.value = ExpenseUiState.Error(e.message ?: "Failed to create expense")
            }
        }
    }

...

}

4.3.9 Penjelasan Penerapan ViewModel

Bentuk tampilan layar diwakili sealed interface ExpenseUiState yang punya turunan Loading, Success, dan Error. State tersebut disimpan di dalam _uiState yang bertipe MutableStateFlow, yaitu wadah satu nilai yang bisa diamati sehingga tiap pergantian nilainya langsung diberitahukan ke pengamatnya. Variabel _uiState dibuat private supaya hanya ViewModel yang berhak mengubahnya, lalu ke luar hanya uiState yang diekspos lewat asStateFlow yang read only. Begitu ViewModel dibuat, blok init langsung memanggil loadExpenses sehingga daftar pengeluaran sudah mulai diambil tanpa menunggu aksi pengguna.

Bagian yang menarik di modul ini ada pada createExpense, karena ia menjaga supaya tidak ada pengeluaran tanpa kas sumber dana. Alurnya bisa diurutkan seperti berikut:
1. Function lebih dulu memeriksa apakah kasId kosong. Bila kosong, _uiState langsung diisi Error lalu function berhenti di situ, sehingga proses tulis sama sekali tidak dijalankan.
2. Bila kasId terisi, prosesnya berlanjut di dalam viewModelScope.launch, yaitu coroutine scope yang umurnya mengikuti ViewModel.
3. ViewModel memanggil repository.createExpense untuk mengirim pengeluaran baru ke Supabase lewat RPC.
4. Begitu pencatatan berhasil, ViewModel memanggil loadExpenses lagi supaya daftar terbaru ikut terbaca.
5. Bila terjadi kegagalan, exception-nya ditangkap lalu _uiState diubah menjadi Error berisi pesannya.

Pemanggilan loadExpenses di akhir itulah yang membuat saldo dan daftar di layar selalu menyesuaikan. loadExpenses akan membaca ulang tabel expense lalu menulis nilai baru ke _uiState, dan karena _uiState bersifat observable, perubahan nilai ini akan mendorong layar menggambar ulang dirinya dengan data terbaru. Pembatalan pengeluaran lewat cancelExpense juga mengikuti pola yang sama, jadi begitu sebuah pengeluaran dibatalkan dan saldo kas dikembalikan oleh database, daftar di layar akan ikut diperbarui.

4.3.11 Penjelasan Kode UI Compose

ExpenseScreen punya keunikan dibanding layar lain, sebab ia menerima dua ViewModel sekaligus yaitu expenseViewModel untuk data pengeluaran dan kasViewModel untuk daftar kas. Hal ini memang diperlukan, karena tiap pengeluaran harus menunjuk ke satu kas sumber dana, dan daftar kas itu dipegang oleh modul kas. Kedua state-nya sama sama diambil lewat collectAsStateWithLifecycle supaya layar ikut menggambar ulang begitu salah satu state-nya berganti.

Ketika showAddSheet bernilai true, layar tidak langsung menampilkan semua kas. Ia lebih dulu menyaring daftar kas dan hanya menyisakan yang aktif lewat filter is_active, sehingga pengguna tidak bisa membebankan pengeluaran ke kas yang sudah dinonaktifkan. Bila kebetulan state kas belum siap, hasil penyaringannya akan jatuh ke emptyList lewat operator elvis supaya layar tetap aman. Daftar yang sudah disaring tadi kemudian dioper ke ExpenseSheet supaya pengguna bisa memilihnya lewat sebuah dropdown. Begitu form dikonfirmasi, tombolnya memanggil createExpense dengan membawa deskripsi, nominal, serta id kas yang dipilih, lalu showAddSheet dikembalikan ke false supaya sheet tertutup. Daftar pengeluarannya sendiri digambar lewat LazyColumn yang tiap barisnya memakai composable ExpenseItem. Pada baris yang belum dibatalkan masih tersedia tombol Cancel, dan tombol itulah yang nantinya memanggil cancelExpense. Baris yang sudah dibatalkan akan tampil lebih redup serta diberi label Cancelled, dan kondisi tampilan ini ditentukan langsung dari property is_cancelled milik model.
